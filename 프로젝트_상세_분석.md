분석일: 2026-04-14

프로젝트명: `content-summary-analyzer` / 글 요약 분석기

## 1. 한 줄 요약

이 프로젝트는 사용자가 로컬 폴더에 넣은 `pdf`, `png`, `jpg`, `jpeg` 파일을 Tauri 데스크톱 앱에서 스캔하고, Gemini API 또는 데모 모드로 요약/분석/카테고리 분류를 수행한 뒤 결과 PDF, Markdown, JSON 메타데이터를 로컬 아카이브 폴더에 저장하는 개인용 문서 요약 분석 도구다.

## 2. 기술 스택

| 영역 | 사용 기술 |
| --- | --- |
| 데스크톱 런타임 | Tauri 2.0 |
| 백엔드 | Rust 2021 |
| 프론트엔드 | React 18, TypeScript, Vite 5 |
| 로컬 DB | SQLite via `rusqlite` bundled |
| AI 연동 | Google Gemini REST API via `reqwest` |
| PDF 생성 | `pdf-lib`, `@pdf-lib/fontkit`, Malgun Gothic font |
| 파일 스캔 | Rust `walkdir` |
| 문서/가이드 생성 | Node ESM script + `pdf-lib` |

## 4. 실행 및 빌드 흐름

`package.json` 기준 주요 명령은 다음과 같다.

| `npm run build` | React/Vite 정적 빌드, 결과는 `dist/` |

| `npm run tauri:check` | Rust `cargo check` 실행 |

Tauri 설정은 `src-tauri/tauri.conf.json`에 있으며 앱 이름은 `글 요약 분석기`, identifier는 `com.contentsummaryanalyzer.app`이다. 프론트엔드 개발 URL은 `http://localhost:1420`, 배포용 프론트엔드 산출물은 `../dist`를 사용한다.

## 5. 핵심 사용자 흐름

1. 사용자가 원본 PDF/이미지 파일을 `data/source-files` 또는 설정한 원본 폴더에 넣는다.
2. React UI가 Tauri command `list_source_files`를 호출한다.
3. Rust 백엔드가 `walkdir`로 폴더 1-depth 파일을 스캔하고 지원 확장자만 반환한다.
4. 기존 DB job 상태와 병합해 화면에 `new`, `processed`, `failed`, `processing` 상태를 표시한다.
5. 사용자가 전체 실행, 선택 분석, 또는 자동 실행을 시작한다.
6. `run_batch`가 대상 파일마다 job을 생성하거나 기존 job을 재사용한다.
7. Gemini API 키가 비어 있으면 데모 분석을 만든다. 키가 있으면 Gemini REST API에 실제 파일을 전송해 분석한다.
8. 프론트엔드가 분석 결과로 Markdown과 결과 PDF를 만든다.
9. `save_generated_artifacts`가 Markdown, PDF, JSON 메타데이터를 로컬 아카이브에 저장하고 DB 경로를 업데이트한다.
10. 최근 결과 목록과 상세 패널에서 결과 PDF와 원본 파일을 열 수 있다.

## 6. 프론트엔드 구조

프론트엔드의 실제 중심은 `app/src/app/App.tsx`다. `app/src/features/*` 아래에도 기능별 패널 파일이 있지만, 현재 주요 UI와 상태 관리는 `App.tsx`에 집중돼 있다.

주요 상태:

- `settings`: 원본 폴더, 아카이브 폴더, 카테고리 텍스트, Gemini 모델, 폴링 주기, 자동 실행 여부
- `sessionApiKey`: 세션 전용 Gemini API key. 저장소에 영구 저장하지 않는 구조다.
- `sourceFiles`: 원본 폴더 스캔 결과
- `recentJobs`: 최근 작업 목록
- `selectedJob`: 결과 상세 표시 대상
- `selectedSourcePaths`: 체크박스로 선택한 신규 파일 목록
- `runSummary`: 배치 실행 처리/스킵/실패 요약

주요 함수:

- `bootstrap`: 설정 로드 후 원본 파일과 최근 작업 갱신
- `refreshSourceFiles`: 원본 폴더 스캔 및 선택 상태 정리
- `pollSourceFiles`: 폴링 기반 자동 실행
- `handleApplyFolders`: 경로/설정 저장
- `handleCreateCategoryFolders`: 카테고리 폴더 생성
- `handleStartBatch`: 전체/선택/자동 배치 분석 실행
- `handleOpenJob`: 결과 상세 조회

Tauri 호출은 `app/src/lib/api.ts`에 모여 있다. 결과 Markdown은 `app/src/lib/markdown.ts`, 결과 PDF 생성과 원본 PDF/이미지 병합은 `app/src/lib/pdf.ts`가 담당한다.

## 7. 백엔드 구조

`src-tauri/src/main.rs`는 Tauri 앱 초기화 시 현재 작업 디렉터리를 기준으로 `Database`를 만들고, 다음 command를 등록한다.

- `load_settings`
- `save_settings`
- `list_source_files`
- `create_category_folders`
- `run_batch`
- `save_generated_artifacts`
- `read_file_base64`
- `list_jobs`
- `get_job_detail`
- `open_path`

`src-tauri/src/commands/mod.rs`는 프론트엔드와 백엔드의 경계다. 배치 실행, 상태 병합, 데모 분석 생성, 파일 base64 읽기, OS 경로 열기 등을 처리한다.

`src-tauri/src/capture/mod.rs`는 원본 폴더 스캔 전용이다. 현재는 `WalkDir::new(...).max_depth(1)`라 하위 폴더 깊은 재귀 분석은 하지 않는다. 지원 확장자는 `pdf`, `png`, `jpg`, `jpeg`다.

`src-tauri/src/gemini/mod.rs`는 Gemini 호출 전용이다.

- PDF는 Gemini Files API의 resumable upload를 사용한다.
- PNG/JPG는 `inline_data` base64 방식으로 보낸다.
- 응답은 JSON만 오도록 프롬프트하고, 실제 응답 문자열에서 첫 `{`부터 마지막 `}`까지 추출해 `AnalysisResult`로 역직렬화한다.
- Gemini가 허용 카테고리 밖의 값을 반환하면 마지막 카테고리로 보정한다.
- PDF 업로드 후 Gemini 파일 삭제를 시도하지만, 삭제 실패는 분석 실패로 취급하지 않는다.

`src-tauri/src/storage/mod.rs`는 SQLite와 파일 산출물 저장을 담당한다.

- DB 파일: `data/content_summary_analyzer.db`
- 과거 DB 이름 `data/thread_likecapture.db`가 있으면 새 이름으로 자동 rename
- 결과 저장 루트: 기본 `data/archive`
- Markdown/JSON 저장: `data/archive/meta`
- 결과 PDF 저장: `data/archive/{category}`

## 8. 데이터 모델과 DB

프론트엔드 타입과 Rust 모델은 대체로 camelCase 직렬화 규칙으로 맞춰져 있다.

핵심 모델:

- `AppSettings`: source/archive folder, categories, Gemini model, polling, auto run
- `SourceFileItem`: 스캔된 원본 파일과 처리 상태
- `AnalysisResult`: title, category, one-line summary, 상세 설명, 핵심 포인트, 인사이트, 불확실성
- `JobListItem`: 목록 표시용 작업 정보
- `JobDetail`: 상세 작업 정보와 분석 JSON
- `BatchRunResult`: processed/skipped/failed 카운트와 처리된 job 목록

DB 테이블:

- `settings(key TEXT PRIMARY KEY, value TEXT NOT NULL)`
- `jobs(...)`

`jobs` 주요 컬럼:

- `id`
- `title`
- `status`
- `category`
- `source_group`
- `source_group_path`
- `created_at`, `updated_at`
- `one_line_summary`
- `source_pdf_path UNIQUE`
- `raw_file_name`
- `output_pdf_path`
- `markdown_path`
- `meta_path`
- `analysis_json`
- `error_message`

스킵 기준은 “기존 job이 있고 `output_pdf_path`가 있으며 해당 PDF 파일이 실제로 존재하는지”다. DB 기록만 있고 PDF가 사라진 경우에는 다시 처리될 수 있게 설계돼 있다.

## 9. 결과 파일 생성 방식

분석 완료 후 프론트엔드가 다음 작업을 수행한다.

- `buildMarkdownReport`: 분석 결과를 Markdown 문서로 변환
- `readFileBase64`: 원본 파일을 Rust에서 base64로 읽기
- `buildPdfBase64`: 요약/분석 페이지를 생성하고 원본 PDF 또는 이미지를 뒤에 붙임
- `saveGeneratedArtifacts`: Markdown/PDF/JSON 메타데이터를 저장

PDF는 `app/src/assets/malgun.ttf`를 임베드해 한글 표시를 처리한다. 원본이 PDF이면 모든 페이지를 복사해 뒤에 붙이고, 이미지이면 새 페이지에 스케일링해서 삽입한다.

파일명은 Rust 저장소 계층에서 만든다. 분석 제목, 한 줄 요약, 원본 파일명에서 한글만 뽑아 최대 12자로 압축하고, 중복이 있으면 숫자 suffix를 붙인다. 한글 후보가 없으면 기본값 `요약결과`를 사용한다.

## 10. 보안 및 개인정보 관점

긍정적인 점:

- Gemini API key는 프론트엔드 세션 상태에만 있으며 DB settings에 저장하지 않는다.
- 브라우저 자동화, 구글 로그인 제어, 쿠키 수집 기능은 코드상 보이지 않는다.
- 실제 외부 전송 대상은 사용자가 지정한 원본 파일과 Gemini 분석 요청이다.
- Gemini PDF 업로드 후 파일 삭제를 시도한다.

주의할 점:

- 원본 파일 내용은 Gemini API로 전송될 수 있으므로 민감 문서 처리 전 사용자 고지가 필요하다.
- API key는 저장하지 않지만 React 상태와 Tauri invoke 인자로는 전달된다.
- `open_path`는 Windows `cmd /C start`를 사용한다. 내부에서 사용자가 입력한 경로를 열기 때문에, 경로 입력 UI가 악성 문자열을 받을 가능성은 계속 주의해야 한다.
- `tauri.conf.json`의 CSP가 `null`이다. 로컬 데스크톱 앱이라도 외부 콘텐츠를 도입할 경우 보안 정책 재검토가 필요하다.

## 12. 확인된 리스크와 개선 포인트

우선순위 높음:

- `@pdf-lib/fontkit`가 lockfile에 고정되어 있지 않다. 현재 로컬 `node_modules`에만 있는 extraneous 의존성이라 재현 가능한 빌드가 깨질 수 있다.
- Gemini 모델 옵션에 `gemini-3.1-flash-lite-preview`가 있다. 실제 사용 가능 여부는 Google API의 현재 모델 제공 상태에 의존하므로, 오류 시 사용자에게 모델 변경 안내가 필요하다.
- Gemini 응답 JSON 추출이 첫 `{`와 마지막 `}` 기준이다. 모델 응답에 JSON 외 텍스트가 섞이는 경우 일부는 복구되지만, 중괄호가 설명문에 섞이면 파싱 리스크가 있다.
- `open_path`가 `cmd /C start` 기반이라 경로 문자열 처리에 보수적인 검증이 필요하다.

우선순위 중간:

- 프론트엔드 UI와 상태 관리가 `App.tsx`에 크게 집중돼 있다. 이미 `features/` 디렉터리가 있으므로 기능별 패널로 분리하면 유지보수성이 좋아진다.
- 원본 폴더 스캔이 `max_depth(1)`이라 하위 폴더 분석 요구가 생기면 확장해야 한다.
- 배치 처리는 파일을 순차 처리한다. 안정성은 좋지만 대량 파일 처리 시 시간이 길어질 수 있다.
- PDF 생성이 프론트엔드에서 수행되므로 큰 PDF를 다룰 때 브라우저 메모리 부담이 커질 수 있다.
- Vite 번들이 1 MB 이상이다. PDF 관련 라이브러리 때문에 큰 것으로 보이며, lazy import나 chunk 분리 검토 가치가 있다.

우선순위 낮음:

- `docs/`와 루트 기획 문서가 이미 존재하지만, 일부 생성 스크립트 출력명/문서명 관리는 더 일관화할 수 있다.
- `data/archive/raw` 폴더는 startup 스크립트에서 만들지만 현재 핵심 저장 로직에서는 직접 쓰지 않는다.
- `JobStatus` 프론트 타입은 `pending | analyzing | completed | failed`인데 `SourceFileItem.status`는 `new | processed | failed | processing`으로 별도 매핑된다. 의도된 구조지만 문서화가 필요하다.

## 13. 기능 상태 평가

완성도가 높은 부분:

- Tauri command 경계가 비교적 명확하다.
- SQLite 기반 작업 이력과 스킵 로직이 있다.
- API key를 영구 저장하지 않는 구조가 반영돼 있다.
- 데모 모드가 있어 API key 없이도 로컬 흐름을 검증할 수 있다.
- 결과 PDF에 분석 페이지와 원본 파일을 병합하는 흐름이 구현돼 있다.
- 기존 DB 이름을 새 DB 이름으로 마이그레이션하는 호환 로직이 있다.

보강하면 좋은 부분:

- 재현 가능한 npm 의존성 고정
- Gemini 모델/응답 실패에 대한 사용자 친화적 오류 표시
- 대용량 PDF 처리 시 진행률과 메모리/시간 처리 전략
- 기능별 React 컴포넌트 분리
- `open_path` 경로 검증
- 배포용 Tauri bundle 설정 정리
- 프론트엔드 테스트 또는 백엔드 단위 테스트 추가

## 14. 추천 작업 순서

1. `@pdf-lib/fontkit`를 `package.json` dependencies에 추가하고 lockfile을 갱신한다.
2. `npm ci` 기준으로 깨끗한 설치 빌드를 확인한다.
3. Gemini 모델 목록을 현재 API에서 실제 사용 가능한 값으로 정리하거나, 실패 시 모델 변경 가이드를 UI에 노출한다.
4. `open_path` 입력을 파일/디렉터리 경로로 제한하고 존재 여부를 검사한다.
5. `App.tsx`를 설정, 실행, 원본 파일 목록, 최근 결과, 상세 결과 패널 단위로 분리한다.
6. PDF 생성 관련 코드를 lazy import로 전환해 초기 JS chunk 크기를 줄인다.
7. `run_batch`, `storage`, `gemini`의 실패 케이스에 대한 테스트를 추가한다.

## 15. 결론

현재 프로젝트는 “로컬 문서 투입 → Gemini/데모 분석 → 카테고리별 결과 PDF와 메타데이터 저장”이라는 핵심 제품 흐름이 이미 구현돼 있고, 프론트엔드 빌드와 Rust 체크도 통과한다. 가장 먼저 해결해야 할 기술적 리스크는 `@pdf-lib/fontkit` 의존성이 `node_modules`에만 존재하고 `package.json`/`package-lock.json`에 고정되어 있지 않은 점이다. 그 다음은 Gemini 응답/모델 실패 처리, `open_path` 경로 검증, 큰 PDF 처리와 번들 크기 개선 순서로 보는 것이 합리적이다.
